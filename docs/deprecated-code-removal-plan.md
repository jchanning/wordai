# Deprecated Code Removal Plan

_Generated by static analysis of the codebase. All deprecated items are located in `WordEntropy.java` and its supporting classes. No deprecated Spring Security patterns were found â€” `SecurityConfig` already uses the modern Lambda DSL._

---

## Executive Summary

The project contains a **dual-path architecture** in `WordEntropy.java` originating from a Phase 1 optimisation that replaced a `ConcurrentHashMap`-based response cache with a dense `ResponseMatrix`. The migration was 80% complete: the new matrix path (`useMatrix = true`) was hard-wired and the pre-computation path was fully updated, but **the legacy path was never deleted**. As a result:

- A `private final boolean useMatrix = true` constant creates permanently dead `else` branches.
- A `static ConcurrentHashMap<WordPairKey, Short> responseCache` is still used at runtime by `getResponseBuckets()`, called by `SelectBellmanFullDictionary` and the lazy-entropy path of `getEntropy()`.
- Eleven distinct deprecated items span three categories: **dead code** (safe to delete), **orphan classes** (delete after callers removed), and **live legacy code requiring migration** (must migrate before removal).

No deprecated stdlib or framework APIs were detected outside `WordEntropy.java` and its direct dependencies.

### DEP-08 â€” `getResponseBuckets(String)` public method _(live, migration required)_

| Field | Value |
|---|---|
| **Location** | `WordEntropy.java:353` |
| **Type** | Public method, not annotated `@Deprecated`, but its only implementation (`computeResponseBuckets`) uses the deprecated `responseCache` + `WordPairKey` |
| **External callers** | `SelectBellmanFullDictionary.java:85` (sole external caller) |
| **Internal callers** | `precomputeWithCache()` (DEP-05, dead) and `calculateEntropy()` â†’ `getEntropy()` (live, lazy path) |

This is the **critical migration item** that gates all downstream removals.

**Migration plan:**

1. **Update `computeResponseBuckets(String guessWord)`** â€” replace the `responseCache`/`WordPairKey` implementation with the `ResponseMatrix`-based logic from DEP-06's `getResponseBucketsFromMatrix()`:
   ```java
   private Map<Short, Set<String>> computeResponseBuckets(String guessWord) {
       int guessId = responseMatrix.getWordId(guessWord);
       Map<Short, Set<String>> result = new HashMap<>();
       int wordCount = responseMatrix.getWordCount();
       for (int targetId = 0; targetId < wordCount; targetId++) {
           short pattern = responseMatrix.getPattern(guessId, targetId);
           String targetWord = responseMatrix.getWord(targetId);
           result.computeIfAbsent(pattern, k -> new HashSet<>()).add(targetWord);
       }
       return result;
   }
   ```
2. The `getResponseBuckets(String)` public signature is **retained** â€” `SelectBellmanFullDictionary` continues to call it unchanged, but it now uses the matrix path internally.

**Action:** Migrate `computeResponseBuckets()` to use `ResponseMatrix` as above. No changes required to callers.

There are two valid options:

**Action (recommended):** Keep `ResponsePattern.java` â€” it is a clean, well-documented class with independent value as a utility. Delete `ResponseCacheTest.java` and replace it with a focused `ResponsePatternTest.java` that tests encode/decode invariants directly.
